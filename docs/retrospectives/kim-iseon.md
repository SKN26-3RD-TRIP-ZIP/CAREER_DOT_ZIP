# 김이선

## STAR-L — 무엇을 했고 무엇을 배웠나

### Situation

CAREER.zip 팀 프로젝트에서 운영자가 회원과 AI 면접 프롬프트를 관리할 수 있는 어드민 패널이 필요했고, 해당 영역의 Backend API부터 Frontend 화면까지 전담했습니다.

### Task

- 회원 목록 조회, 검색, 필터, 상태 변경, 삭제 및 초대 기능 구현
- 지표 카드, 주간 면접 세션 차트, 시스템 상태를 포함한 운영 대시보드 구현
- 서비스 운영 중 프롬프트를 교체하거나 이전 버전으로 되돌릴 수 있는 관리 기능 구현
- 관리자에 의해 수행되는 주요 변경 작업의 감사 로그 기록

### Action

- **감사 로그 설계**  
  생성, 수정, 삭제, 상태 변경, 활성화, 롤백 등 6종의 관리자 작업을 단일 테이블에서 관리하도록 설계했습니다. 관리자 계정이 삭제된 후에도 기록이 유지되도록 `actor`에 `SET_NULL`을 적용하고, `actor_email`을 스냅샷으로 저장했습니다. 또한 `target_type`, `target_id`를 활용해 변경 대상을 추적하고, 최근에는 대상자 이름인 `target_name`을 함께 노출해 로그의 가독성을 높였습니다.

- **회원 관리의 동시성 안전성 확보**  
  회원 상태 변경과 삭제 과정에 `transaction.atomic()`과 `select_for_update()`를 적용해 동시에 여러 요청이 발생하더라도 데이터 정합성을 유지하도록 구현했습니다.

- **3-tier 프롬프트 관리 구조 구현**  
  `PersonaConfig → PromptTemplate → PromptVersion`으로 이어지는 3계층 구조를 구성했습니다. 활성 버전 포인터를 변경하는 방식으로 재배포 없이 프롬프트를 교체하거나 롤백할 수 있도록 했으며, 진행 중인 면접 세션은 시작 당시의 프롬프트 버전을 유지하도록 설계했습니다. 운영 피드백을 반영해 불필요한 `pressure` 페르소나를 마이그레이션으로 제거하고, `coach`, `practical`, `verifier`의 3종 구조로 정리했습니다.

- **대시보드 지표 고도화**  
  단순 누적 회원 수뿐 아니라 전월 대비 회원 증감률을 계산해 운영자가 변화 추이를 확인할 수 있도록 했습니다. 직전 달 회원 수가 0명인 경우에는 잘못된 비율이 표시되지 않도록 별도 예외 처리를 적용했습니다.

### Result

- 회원 관리, 운영 대시보드, 프롬프트 관리, 감사 로그 등 4개 도메인의 Backend API와 Frontend 화면을 동작 가능한 상태로 완성했습니다.
- 페르소나, 프롬프트 템플릿, 프롬프트 버전으로 구성된 3-tier 관리 구조와 6종의 감사 로그 액션 타입을 구축했습니다.
- 초기 기능 완성 이후에도 감사 로그 대상자 표시, 대시보드 증감률, 페르소나 옵션 정리 등 운영 관점의 개선을 지속해서 반영했습니다.
- 셀프 리뷰를 통해 감사 로그 조회의 N+1 가능성, 회원 삭제 로직의 중복, 일부 생성 로직의 원자성 등 후속 개선 과제도 발견했습니다.

### Learned

- 기능을 새롭게 추가하는 것뿐 아니라 불필요한 요소를 제거하고 구조를 단순화하는 작업도 중요한 개선이라는 점을 배웠습니다.
- 운영 지표는 단순한 숫자보다 전월 대비 변화처럼 비교할 수 있는 맥락이 함께 제공될 때 의사결정에 더 유용하다는 것을 확인했습니다.
- 사용자 편의를 위한 기능을 구현할 때에도 데이터 조회 비용과 확장 가능성을 함께 고려해야 한다는 점을 배웠습니다.

## 5 Whys — 반복된 문제의 근본 원인

### 문제

기능을 추가할 때마다 N+1 쿼리, 중복 로직, 권한 처리 불일치와 같은 유지보수 문제가 함께 발생했습니다.

1. **왜 추가 쿼리가 발생했는가?**  
   감사 로그의 대상자 이름을 표시하기 위해 각 로그마다 사용자를 개별 조회했기 때문입니다.

2. **왜 개별 조회 방식으로 구현했는가?**  
   기능을 빠르게 완성하기 위해 직렬화 메서드 내부에서 간단하게 조회하는 방식을 선택했기 때문입니다.

3. **왜 성능 최적화를 뒤로 미뤘는가?**  
   개발 단계에서는 데이터가 많지 않아 성능 문제가 즉시 체감되지 않았고, 대량 데이터 상황의 비용을 별도로 측정하지 않았기 때문입니다.

4. **왜 쿼리 비용을 측정하지 않았는가?**  
   `assertNumQueries`나 쿼리 로깅과 같은 성능 검증 절차가 개발 습관과 리뷰 과정에 포함되어 있지 않았기 때문입니다.

5. **왜 검증 절차가 마련되지 않았는가?**  
   기능 구현을 우선하면서 완료 기준에 쿼리 수, 중복 로직, 원자성 검토를 포함하지 않았기 때문입니다.

### 근본 원인과 개선 방향

기능 완료의 기준이 정상 동작 여부에 집중되어 있었고, 성능과 유지보수성 검토가 명시적인 완료 조건에 포함되어 있지 않았습니다.

향후에는 다음 항목을 Definition of Done에 포함해 같은 문제의 반복을 줄이고자 합니다.

- 반복 조회와 N+1 쿼리 여부 확인
- 유사한 서비스·View 로직의 중복 여부 확인
- 여러 데이터를 변경하는 작업의 트랜잭션 적용 여부 확인
- 권한 클래스와 API 정책의 일관성 확인
- 주요 조회 API의 쿼리 수 또는 성능 로그 점검
