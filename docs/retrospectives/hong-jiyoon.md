# 홍지윤

## Keep — 좋았던 점과 계속 이어가고 싶은 것

### 1. 기능을 개별 단위가 아니라 전체 흐름으로 바라본 점

JD 분석, 이력서 매칭, 질문 생성, STAR 답변은 각각 독립된 기능처럼 보이지만 실제로는 이전 단계의 결과가 다음 단계의 입력으로 이어지는 구조였습니다.

한 기능의 동작 여부만 확인하는 데서 멈추지 않고, 다음 단계에서 해당 결과가 어떤 형태로 사용되는지 지속적으로 확인하며 구현했습니다. 이를 통해 개별 기능은 정상인데 전체 서비스 흐름에서는 연결되지 않는 문제를 줄이고자 했습니다.

### 2. 막연한 AI 기능을 구체적인 단위로 분리한 점

LLM에 여러 작업을 한 번에 요청하기보다 다음과 같이 역할을 나누어 서비스 함수로 분리했습니다.

- JD 키워드 추출
- 채용 요구사항 분류
- 이력서와 JD의 매칭 점수 계산
- 지원자 역량과 요구사항 사이의 Gap 산출
- 분석 결과를 활용한 예상 질문 생성
- STAR 구조 기반 답변 지원

기능을 세분화하면서 각 단계의 입력과 출력을 명확히 할 수 있었고, 오류가 발생했을 때 원인이 되는 지점을 더 빠르게 찾을 수 있었습니다.

### 3. Submodule 구조에 맞춰 작업 경계를 유지한 점

Frontend와 Backend가 분리된 Submodule 구조에서 담당 영역의 책임 범위를 명확히 하고, 연동 전에 응답 형식과 필요한 데이터 구조를 먼저 맞추려 했습니다.

이를 통해 다른 영역의 코드를 불필요하게 수정하지 않으면서도, Frontend와 Backend가 연결되는 과정에서 발생할 수 있는 혼선을 줄일 수 있었습니다.

## Problem — 아쉬웠던 점과 개선이 필요한 부분

### 1. 구현된 코드와 실제 사용되는 코드를 동일하게 판단한 점

코드가 작성되어 있다는 이유만으로 해당 기능이 실제 서비스에서 사용되고 있다고 판단한 부분이 있었습니다.

예를 들어 GitHub 저장소에서 코드 조각을 추출하고 분석하는 로직을 구현했지만, 실제 질문 생성 흐름에서는 GitHub README 텍스트만 근거로 사용하고 있었습니다. 함수는 존재했지만 현재 사용자 흐름에서 호출되지 않아 결과적으로 죽은 코드가 남았습니다.

이 경험을 통해 기능 완료 여부는 코드 존재 여부가 아니라 실제 요청 흐름에서 호출되고 결과가 다음 단계까지 전달되는지를 기준으로 판단해야 한다는 점을 배웠습니다.

### 2. 검증과 예외 처리를 뒤로 미룬 점

핵심 기능을 먼저 연결하는 데 집중하면서 다음과 같은 경계값과 예외 상황의 처리가 늦어졌습니다.

- 빈 입력값이 그대로 전달되는 경우
- 특정 값이 없으면 건너뛰어야 하지만 오류가 발생하는 경우
- 일부 분석 결과가 없을 때 다음 단계가 중단되는 경우
- 정상 데이터만을 전제로 작성된 로직

데모에서는 정상 입력을 기준으로 기능이 동작했지만, 실제 QA 관점에서는 예외 상황까지 처리되어야 기능이 완성된다는 점을 확인했습니다.

### 3. 문서와 실제 코드 상태가 어긋난 점

개발 과정에서 기능이 빠르게 변경되면서 문서에 기록된 구현 상태와 실제 코드가 일치하지 않는 경우가 생겼습니다.

미구현으로 알고 있던 기능이 이미 구현되어 있거나, 반대로 완료된 것으로 기록된 기능이 실제 사용자 흐름에서는 호출되지 않는 경우도 있었습니다. 이로 인해 현재 상태를 설명할 때 불확실성이 생겼고, 협업 과정에서 불필요한 오해가 발생할 가능성이 있었습니다.

## Try — 문제를 개선하기 위해 시도한 것

### 1. QA 항목과 실제 코드를 일대일로 대조했습니다

34개의 QA 항목을 기준으로 각 기능의 상태를 다음과 같이 분류했습니다.

- 정상 구현
- 부분 구현
- 미구현

각 판단에는 관련 파일과 코드 위치를 함께 기록했습니다. 이를 통해 기억이나 추측이 아니라 실제 코드 근거를 바탕으로 현재 구현 상태를 설명할 수 있게 되었습니다.

### 2. 미구현 기능과 죽은 코드를 명확하게 구분했습니다

애매한 상태로 남겨두지 않고 다음과 같이 구체적으로 기록했습니다.

- 함수는 존재하지만 실제 사용자 흐름에서 호출되지 않음
- 일부 로직만 구현되었고 후속 단계와 연결되지 않음
- Mock 데이터에서만 동작함
- 정상 입력에서는 동작하지만 예외 처리가 부족함

이를 통해 후속 작업자가 코드를 다시 조사하는 시간을 줄이고, 유지·보완·삭제 중 어떤 결정을 내려야 하는지 판단할 수 있는 근거를 남겼습니다.

### 3. 검증 로직을 기능과 함께 구현하는 방식으로 순서를 바꾸고자 했습니다

앞으로는 핵심 로직을 구현한 뒤 예외 처리를 별도로 추가하는 방식이 아니라, 처음부터 정상 케이스와 실패 케이스를 함께 정의하고 구현하고자 합니다.

기능별로 다음 항목을 함께 확인하는 습관을 만들고자 합니다.

- 정상 입력
- 빈 입력과 누락 값
- 외부 API·LLM 호출 실패
- 이전 단계 결과가 없는 경우
- 실제 사용자 흐름에서의 호출 여부
- 다음 단계로 결과가 정상 전달되는지
- 문서와 코드의 구현 상태가 일치하는지

## 프로젝트를 마치며

이번 프로젝트를 통해 기능이 구현되어 있다는 사실만으로는 서비스가 완성되지 않는다는 점을 배웠습니다. 각 기능이 실제 사용자 흐름에서 호출되고, 이전 단계의 결과가 다음 단계로 정확히 전달되며, 정상 상황뿐 아니라 예외 상황에서도 예측 가능한 방식으로 동작해야 비로소 하나의 기능이 완성됩니다.

앞으로는 AI 기능을 작은 단위로 분리하고 전체 데이터 흐름을 함께 확인했던 강점은 유지하되, 구현과 동시에 예외 처리와 실제 호출 여부를 검증하고 문서의 상태도 지속적으로 최신화하는 개발 방식을 이어가고자 합니다.
