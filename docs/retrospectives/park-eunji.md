# 박은지

## 담당 파트

면접 답변 평가 파이프라인과 최종 리포트 생성 기능, 리포트 화면 연동을 담당했습니다. 사용자가 면접을 마친 이후의 모든 단계가 제 영역이었습니다.

```text
답변 저장
→ 4축 평가 (BEI·CBI·Speech·Grounding)
→ 강점·약점 태그 추출
→ 세션 전체 집계
→ 면접관 페르소나 가중치 적용
→ 질문별 개선 액션 생성
→ 최종 리포트 (조회·PDF·공유 링크)
```

답변 하나는 BEI(STAR 행동 경험 구조), CBI(역량 행동지표), Speech(발화 유창성), Grounding(수치·기술 근거 제시 여부)의 4개 축으로 점수화되며, 세션이 끝나면 이 결과를 면접관 페르소나별 가중치로 재산출해 최종 리포트를 생성합니다. 이 파이프라인의 핵심 단계 대부분이 LLM 호출에 의존하기 때문에, "LLM은 언제든 실패할 수 있고, 같은 입력에도 다른 출력을 낼 수 있다"는 전제가 설계 전체를 관통하는 조건이었습니다.

## Keep — 좋았던 점과 계속 이어가고 싶은 것

### 1. "AI가 실패해도 리포트는 반드시 나온다"를 설계 원칙으로 세운 점

면접을 끝까지 진행한 사용자에게 리포트가 나오지 않는 것이 이 파트의 최악의 실패라고 판단하고, 실패 지점별로 방어선을 두었습니다.

- **답변 단위 예외 격리**: 세션 평가 시 답변마다 개별적으로 예외를 처리해, 한 답변의 평가가 실패해도 나머지 답변은 계속 평가되도록 했습니다. 처리 결과는 `evaluated / skipped / failed` 통계로 반환해 어디까지 성공했는지 추적할 수 있게 했습니다.
- **개선 액션 2단 구조**: 질문별 개선 제안은 약점 태그 기반 템플릿을 먼저 채워두고, LLM 배치 호출 결과로 덮어쓰는 구조로 만들었습니다. LLM이 실패해도 템플릿이 남아 리포트가 빈 채로 나가지 않습니다.
- **전부 실패 시의 동작 정의**: 평가가 모두 실패한 리포트는 저장하지 않고 재시도 가능함을 명시한 503 응답(`retryable: true`)을 반환해, 프론트가 다시 요청하도록 했습니다. 잘못된 리포트를 보여주는 것보다 "다시 시도해 주세요"가 낫다고 판단했습니다.
- **real mode와 mock mode의 분리**: 테스트용 mock 환경에서만 폴백 결과를 반환하고, 운영에서는 OpenAI 실패 시 예외를 그대로 전파해 상위에서 격리하도록 했습니다. 운영에서 실패가 조용히 0점으로 바뀌는 것을 막기 위한 결정이었습니다.

### 2. 멱등성과 동시성을 처음부터 챙긴 점

평가는 몇 번을 다시 돌려도 같은 결과여야 한다고 정의했습니다.

- 이미 평가가 존재하는 답변은 재평가하지 않고 그대로 반환하도록 해, 리포트 재생성이나 백필을 반복해도 중복 평가와 점수 변동이 생기지 않도록 했습니다.
- 평가 결과와 태그 저장은 트랜잭션으로 묶어 원자성을 보장했습니다.
- 동시에 리포트 생성 요청이 두 번 들어오는 경우에는 세션 락(`select_for_update`) 후 재조회하는 방식으로 중복 리포트 생성을 방지했습니다.
- 백필의 멱등성과 실패 격리는 별도 테스트로 검증해, "다시 돌려도 안전하다"를 코드가 아니라 테스트가 보장하도록 했습니다.

### 3. LLM에 맡길 일과 맡기지 않을 일을 구분한 점

모든 것을 LLM에 맡기지 않고, 각 지표의 성격에 맞는 처리 방식을 선택했습니다.

- **발화 유창성은 로컬 분석**: 필러워드·휴지 분석은 형태소 분석기(Kiwi) 기반 로컬 처리로 구현했습니다. 단음절 필러워드("어", "음")를 토큰 정확일치로만 카운트해 "들어", "마음" 같은 오탐을 막았고, 비용이 들지 않으며 결과가 결정적이라 재현성이 보장됩니다.
- **비기술 질문은 grounding 호출 생략**: 기술 근거 검증이 의미 없는 질문 유형에서는 grounding LLM 호출을 아예 생략해 비용과 지연을 줄였습니다.
- **기술 깊이는 SBERT로 보정**: LLM이 개념 점수는 잘 주지만 실제 기술 깊이 판단에는 약해서, 기술 질문에 한해 답변을 사용자 자료와 임베딩 유사도로 비교하는 하이브리드 점수를 적용했습니다. 모델 로드 실패 시에는 기존 LLM 점수를 유지하도록 폴백을 두었습니다.
- **모델 선택은 벤치마크로**: 채점용 LLM은 감으로 고르지 않고 모델 비교 벤치마크를 돌려 선택 근거를 문서로 남겼습니다.

### 4. 설계 근거를 문서로 남긴 점

점수 체계 가이드와 리포트 summary 포맷 문서를 작성해 프론트 연동의 기준으로 삼았습니다. "왜 Speech 점수 상한이 80인가(전달력이 완벽해도 답변 내용을 대체할 수 없다)", "왜 페르소나마다 가중치가 다른가(면접관이 판단하려는 것이 다르다)" 같은 설계 결정의 이유를 함께 기록해, 코드만으로는 보이지 않는 의도를 다른 팀원이 확인할 수 있게 했습니다. 페르소나 가중치는 합이 1.0인지 모듈 로드 시점에 검증하도록 해, 잘못된 수정이 배포 전에 드러나도록 했습니다.

## Problem — 아쉬웠던 점과 개선이 필요한 부분

### 1. 실패에 대응하는 설계에 비해 실패를 줄이는 설계가 부족했던 점

폴백과 격리는 갖췄지만, 실패 확률 자체를 낮추는 쪽은 미완으로 남았습니다. 평가 LLM 3-chain 병렬 호출의 타임아웃이 15초 고정이라 worst-case 지연이 그대로 사용자에게 노출되고, 동시 세션이 많아질 때의 OpenAI rate limit 대응(백오프, 재시도 간격 조절)은 구현하지 못했습니다. 트래픽이 적은 시연 환경에서는 문제가 되지 않았지만, 실제 운영 규모에서는 가장 먼저 드러날 약점이라고 생각합니다.

### 2. 다른 파트와 맞닿는 설정의 정합성 확인이 늦었던 점

페르소나 가중치 설정에 새 페르소나를 추가했을 때, 다른 파트에서는 같은 페르소나를 제거하는 방향으로 마이그레이션이 진행 중이었습니다. 결과적으로 코치형·실무형·검증형 3종으로 정리되었지만, 공유 설정을 변경하기 전에 관련 파트의 작업 방향을 먼저 확인했다면 반대 방향의 작업이 공존하는 기간 자체가 없었을 것입니다. 제 코드 안에서만 판단이 닫혀 있었던 점이 아쉽습니다.

### 3. LLM 출력 스키마에 대한 방어가 사후적이었던 점

JSON 출력을 강제해도 키 누락은 발생할 수 있어 기본값 처리로 방어했지만, 이 방식에는 "키가 누락되면 조용히 0점이 된다"는 부작용이 있습니다. 실패가 명시적으로 드러나는 503과 달리, 스키마 누락은 잘못된 점수가 정상 점수처럼 보이는 형태라 더 위험합니다. 누락 발생 빈도를 기록하고 알리는 관측 장치를 만들지 못하고 모니터링 필요 항목으로만 남긴 점이 아쉽습니다.

### 4. 평가 품질을 정량적으로 확인할 장치가 부족했던 점

모델 선택 시점에는 벤치마크를 돌렸지만, 운영 중 평가 품질의 변동(같은 답변에 대한 점수 산포, 축별 점수 분포의 쏠림 등)을 상시 확인할 수 있는 장치는 없었습니다. "이 점수를 믿어도 되는가"라는 질문에 배포 이후에도 계속 답할 수 있어야 평가 기능이라고 말할 수 있다는 점에서, 관측 가능성은 부가 기능이 아니라 평가 파이프라인의 일부였어야 했습니다.

## Try — 앞으로 시도할 것

### 1. 외부 API 호출 설계에 백오프·재시도 정책을 기본으로 포함하겠습니다

폴백은 실패한 뒤의 대응입니다. 앞으로는 rate limit 백오프, 재시도 간격, 타임아웃의 단계적 조정까지 호출 설계 단계에서 함께 정의해, 실패에 대응하는 설계와 실패를 줄이는 설계를 한 쌍으로 다루겠습니다.

### 2. 공유 자원을 변경할 때는 코드보다 확인을 먼저 하겠습니다

페르소나, 프롬프트, 공통 설정처럼 여러 파트가 함께 쓰는 자원을 변경할 때는, 수정 전에 관련 마이그레이션과 문서 상태를 대조하고 담당자와 방향을 맞추는 것을 작업 순서의 첫 단계로 두겠습니다.

### 3. 조용한 실패를 시끄러운 실패로 바꾸겠습니다

LLM 출력의 스키마 검증 실패를 기본값으로 덮는 데서 멈추지 않고, 발생 사실을 명시적으로 기록하고 빈도를 확인할 수 있는 구조로 만들겠습니다. 잘못된 결과가 정상처럼 보이는 것이 가장 비싼 실패라는 것을 이번 프로젝트에서 배웠습니다.

### 4. 관측 가능성을 기능과 함께 설계하겠습니다

점수 분포, 축별 편차, 실패율 같은 평가 품질 지표를 처음부터 설계 항목에 포함해, 기능이 동작하는지뿐 아니라 결과를 신뢰할 수 있는지를 운영 중에도 확인할 수 있게 하겠습니다.

## 프로젝트를 마치며

이번 프로젝트에서 가장 크게 배운 것은 AI 기능의 완성도는 잘 될 때의 품질이 아니라 실패할 때의 동작으로 결정된다는 점입니다. 폴백, 멱등성, 예외 격리, 실패 시 응답 정의처럼 눈에 보이지 않는 설계가 결국 사용자에게 "면접을 마치면 항상 리포트가 나오는 서비스"라는 신뢰로 이어진다는 것을 경험했습니다.

동시에 평가라는 추상적인 문제를 구체적인 점수 체계로 옮기는 과정도 의미가 컸습니다. 점수 상한과 가중치 하나하나에 "왜"를 붙여 근거를 만들면서, 프롬프트를 작성하는 일보다 평가 철학을 코드와 문서로 옮기는 일이 이 파트의 본질이라는 것을 알게 되었습니다.

앞으로는 실패에 대응하는 설계는 유지하되, 실패를 줄이는 설계와 결과를 신뢰할 수 있는지 확인하는 관측까지 함께 갖춘 개발자가 되고자 합니다.

> 평가 기능의 완성은 점수를 내는 것이 아니라, 그 점수를 믿을 수 있게 만드는 것이다.
