# 박소윤

## STAR-L — 무엇을 했고 무엇을 배웠나

### Situation

CAREER.zip은 IT 구직자의 JD, 이력서, 자기소개서, 프로젝트 정보를 분석해 개인화된 AI 모의면접과 평가 리포트를 제공하는 서비스입니다.

6인 팀의 PM이자 Backend·Frontend·DB 담당으로서 프로젝트 기획과 일정 관리, API 명세 통합, QA 관리뿐 아니라 회원가입·로그인·인증, 사용자 프로필과 지원 자료 입력 API, 인증·입력 화면과 API 연동, DB 설계 및 데이터 전처리, 마이페이지·포인트·가드레일 고도화, AWS 배포 설계까지 폭넓게 담당했습니다.

서비스는 다음과 같은 흐름으로 구성되었습니다.

```text
회원가입·로그인
→ 사용자 정보 및 지원 자료 입력
→ AI 분석
→ 면접 설정
→ 질문 생성과 답변
→ 평가
→ 최종 리포트
→ 마이페이지
```

프로젝트가 진행될수록 제 담당 기능뿐 아니라 다른 팀원의 분석, 질문 생성, 면접, 평가, 리포트 기능까지 이해해야 전체 서비스 흐름의 문제를 판단할 수 있었습니다.

### Task

프로젝트에서 해결해야 했던 핵심 과제는 다음과 같았습니다.

- 사용자 입력 데이터가 분석, 질문 생성, 면접, 평가, 리포트까지 끊김 없이 전달되도록 전체 데이터 흐름 연결
- 여러 팀원이 병렬 개발하면서 발생한 API 명세와 실제 구현의 불일치 해소
- 단발성 면접 기능을 반복 연습과 성장 확인이 가능한 서비스로 고도화
- 정상 입력뿐 아니라 파일 손상, 인증 실패, AI 호출 실패, 악의적 입력 등 실패 상황 설계
- 기능 구현 상태와 실제 검증 완료 상태를 구분하고 재현 가능한 QA 기준 마련
- 개발 환경에서 동작한 기능이 AWS 배포 환경에서도 동일하게 작동하도록 통합 구조 설계

### Action

##### 1. API와 데이터 구조를 먼저 정리했습니다

팀 전체 API를 10개 영역으로 나눈 통합 명세를 작성하고, 인증, 프로필, JD, 이력서, 자기소개서, 프로젝트 영역의 API 계약을 구체화했습니다.

Endpoint와 Request·Response뿐 아니라 다음 항목까지 명세에 포함했습니다.

- JWT 인증 방식
- 에러 상태와 응답 구조
- 파일 형식과 크기 제한
- 사용자별 데이터 소유권 검증
- 상태값과 `null` 처리
- 프론트엔드와 백엔드의 역할 범위

DB는 이후의 마이그레이션 비용을 고려해 서비스 흐름에 필요한 필드를 초기 설계에 반영했습니다. 또한 AI 분석에 사용할 데이터의 수집 기준과 전처리 방식을 정리해 입력 단계부터 분석 파이프라인까지 데이터 품질을 관리했습니다.

이를 통해 API 명세는 단순한 결과 문서가 아니라 프론트엔드, 백엔드, 각 기능 담당자 사이의 개발 계약이라는 점을 배웠습니다.

##### 2. 회원가입과 이메일 인증을 실제 동작 기준으로 재구성했습니다

초기 회원가입 구조에서는 이메일 인증 전에 `User`가 생성되는 문제가 있었습니다. 이를 가입 정보를 임시 저장하는 `PendingRegistration` 구조로 변경하고, 6자리 인증코드 확인 후 실제 사용자를 생성하도록 재구성했습니다.

다음 예외 상황도 함께 처리했습니다.

- 인증코드 유효시간
- 재발송 쿨다운
- 인증 시도 횟수 제한
- 미인증 이메일 재가입 시 인증코드 재발송
- 인증 완료 이메일 재가입 차단
- 실제 SMTP 발송 성공 여부 반영

프론트엔드에서는 인증·입력 화면에 남아 있던 Mock 데이터를 실제 로그인 사용자 정보로 교체했습니다. 또한 `localhost`와 `127.0.0.1` 혼용으로 Refresh Cookie가 저장되지 않던 문제를 추적해 해결하고, 여러 요청에서 동시에 401이 발생해도 Token Refresh가 한 번만 실행되도록 정리했습니다.

이 과정을 통해 화면에 값이 보이는 것과 실제 사용자 데이터가 연결되어 있는 것은 전혀 다른 상태라는 점을 체감했습니다.

##### 3. 구현 여부가 아니라 테스트 결과를 기준으로 판단했습니다

기능이 작성되었다는 설명만으로 완료를 판단하지 않고, 실제 테스트와 E2E 결과를 기준으로 상태를 관리했습니다.

프로젝트 후반에는 Backend 전체 756개 테스트 통과를 확인했으며, 면접·리포트·평가 등 앱별 테스트와 Frontend 테스트·Build도 반복 검증했습니다.

테스트 과정에서 다음과 같은 숨은 문제도 발견했습니다.

- 리포트의 `null` 점수가 0점으로 표시되는 Adapter 결함
- OpenAI 호출 실패 시 Mock 질문이나 0점 리포트로 조용히 대체되는 문제
- 문서와 실제 API 경로의 불일치
- 구현되었지만 Frontend와 연결되지 않은 기능

기능 상태도 단순한 PASS가 아니라 다음과 같이 구분했습니다.

```text
PASS
FAIL
BLOCKED
NEEDS_CONFIRMATION
NOT_IMPLEMENTED
MOCK_ONLY
```

이를 통해 코드 작성 완료, 단위 테스트 통과, 통합 검증, 실환경 E2E 성공은 서로 다른 단계라는 점을 배웠습니다.

##### 4. 정상 흐름뿐 아니라 실패 상황을 설계했습니다

문서 업로드에는 PDF·DOCX 형식 검증과 Parsing, 이미지 입력의 OCR 분기 처리를 적용했습니다.

면접 입력에는 등급 기반 가드레일을 설계했습니다.

- 정상 입력 허용
- 개인정보·비밀정보 탐지
- 주민등록번호 등 민감정보 차단
- XSS·SQL Injection 탐지
- Prompt Injection과 내부 지침 유출 방지
- 유해·불법 콘텐츠 차단
- 자동 제재 대신 관리자 검토가 필요한 이벤트 분리

정상 입력에서 기능이 작동하는 것뿐 아니라, 잘못된 입력과 악의적 입력에서 어떻게 안전하게 실패할지까지 설계하는 것이 서비스 개발이라는 점을 배웠습니다.

##### 5. 단발성 면접이 아닌 성장 서비스로 고도화했습니다

마이페이지를 단순 면접 기록 목록이 아니라 사용자의 변화와 약점을 확인할 수 있는 성장 대시보드로 발전시켰습니다.

주요 고도화 항목은 다음과 같습니다.

- 면접별 점수 변화
- 반복적으로 나타난 약점 태그 TOP 3
- 세션 전반부와 후반부 비교
- 개선·유지·하락 상태 판정
- 행동 기반 포인트 적립·차감
- 중복 적립 방지와 환불 정책
- 약관 동의 이력
- 휴면 및 탈퇴 정책

새로운 아이디어를 모두 구현하기보다 P0, P1, P2로 우선순위를 구분해 필수 기능과 후속 확장 기능을 관리했습니다.

##### 6. 배포를 전체 서비스의 통합 문제로 바라봤습니다

AWS EC2, Docker, Nginx, Aiven DB, S3를 연결한 배포 구조를 설계하고, 배포 실행과 QA Gate, Migration 승인 등 역할 경계를 문서화했습니다.

배포 과정에서는 다음 항목을 함께 확인했습니다.

- `DEBUG=False`
- `ALLOWED_HOSTS`
- HTTPS와 Secure Cookie
- 환경변수 관리
- DB 접근 허용 목록
- Migration 실행 순서
- Nginx Reverse Proxy
- Frontend·Backend Container 연결

`.env.example`과 실제 코드의 환경변수 사용처를 대조해 사용하지 않는 Key를 제거하고, 비밀값은 문서나 저장소에 노출하지 않는 원칙을 유지했습니다.

배포는 개발 마지막에 별도로 추가하는 작업이 아니라, 지금까지 만든 모든 기능이 실제 환경에서도 같은 방식으로 동작하는지 확인하는 최종 통합 테스트에 가깝다는 점을 배웠습니다.

##### 7. 문제가 발생했을 때 작업물 보존을 우선했습니다

개발 중 `git index.lock`, 브랜치 동기화 충돌, CRLF 차이에 따른 대량 변경 표시와 같은 문제가 발생했습니다.

이때 바로 Reset하거나 Commit하지 않고 다음 순서로 대응했습니다.

1. `git status`로 현재 상태 확인
2. 실제 코드 변경과 환경 아티팩트 구분
3. 원본 작업 보존 여부 확인
4. 충돌과 줄바꿈 문제 해결
5. 변경 범위 재검토 후 Commit

Git 문제는 빠르게 명령을 실행하는 것보다 현재 상태를 정확히 파악하고 원본을 보존하는 것이 먼저라는 점을 배웠습니다.

### Result

- 회원가입, 이메일 인증, 로그인, Token 관리와 실제 사용자 정보 연결을 완성했습니다.
- JD, 이력서, 자기소개서, 프로젝트 데이터를 저장하고 AI 분석 파이프라인으로 전달하는 기반을 구축했습니다.
- DB 설계와 데이터 전처리를 통해 서비스 데이터의 구조와 품질을 관리했습니다.
- API 명세와 실제 구현을 대조하며 인증, 문서 업로드, 면접, 리포트 영역의 통합 문제를 해결했습니다.
- Backend 756개 테스트 통과와 Frontend 테스트·Build 결과를 기준으로 프로젝트 상태를 검증했습니다.
- 성장 대시보드, 포인트, 가드레일 등 서비스 고도화 방향을 구체화했습니다.
- 요구사항, WBS, API 명세, ERD, 화면설계서, 94개 QA Test Case, 테스트 보고서, 배포 설계 등 프로젝트 산출물을 정리했습니다.

가장 큰 결과는 시각의 변화였습니다. 초반에는 제 담당 기능을 정상적으로 구현하는 것이 목표였다면, 후반에는 서비스 전체의 데이터 흐름, 기능 간 계약, 테스트 상태, 배포 가능성까지 함께 판단하게 되었습니다.

### Learned

- **구현보다 먼저 인터페이스를 합의해야 합니다.**  
  URL, 필드명, 상태값, 인증 방식, 사용자 생성 시점이 조금만 달라도 개별 기능은 정상인데 전체 서비스는 동작하지 않을 수 있습니다.

- **완료와 검증 완료는 다릅니다.**  
  코드 작성, 로컬 성공, 단위 테스트, 통합 테스트, E2E, 실환경 검증은 서로 다른 상태로 관리해야 합니다.

- **서비스 품질은 실패 상황에서 더 명확하게 드러납니다.**  
  손상 파일이나 AI 호출 실패가 조용히 잘못된 결과로 바뀌지 않도록 명확한 오류와 재시도 경로가 필요합니다.

- **문서화는 개발 이후의 기록이 아니라 개발 과정의 도구입니다.**  
  명세와 QA 문서를 작성하면서 API 불일치, 미검증 기능, 모호한 담당 범위를 발견할 수 있었습니다.

- **혼자 많이 처리하는 것과 팀 생산성을 높이는 것은 다릅니다.**  
  모든 문제를 직접 해결하기보다 담당자에게 조건과 검증 기준을 명확히 전달하고 결과를 확인하는 방식이 필요하다는 점을 배웠습니다.

## 5 Whys — 통합 검증이 후반까지 남은 이유

### 문제

전체 E2E 검증과 실제 배포 환경에서의 확인이 프로젝트 후반까지 남았습니다.

1. **왜 전체 E2E 검증이 늦어졌는가?**  
   각 담당자가 자신의 기능 구현을 우선했고, 전체 서비스 연결은 개별 기능이 어느 정도 완성된 후에 확인했기 때문입니다.

2. **왜 전체 연결을 더 일찍 확인하지 못했는가?**  
   API와 데이터 구조가 프로젝트 중간에도 변경되었고, 담당자마다 기능 확정 시점이 달랐기 때문입니다.

3. **왜 API와 데이터 구조 변경이 반복되었는가?**  
   예외 상황, Frontend 연동 방식, 인증 정책, 비동기 처리 방식을 충분히 확정하지 않은 상태에서 병렬 개발을 시작했기 때문입니다.

4. **왜 충분한 합의 전에 개발을 시작했는가?**  
   제한된 일정에서 문서 설계보다 코드 작성이 더 빠른 진척처럼 느껴졌기 때문입니다.

5. **근본적으로 무엇이 부족했는가?**  
   전체 사용자 시나리오를 기준으로 한 조기 통합 계획과 명확한 Definition of Done이 부족했습니다.

### 개선 방향

다음 프로젝트에서는 초반부터 최소 데이터로 대표 E2E 흐름을 먼저 연결하고, 이후 각 기능을 확장하고자 합니다.

```text
회원가입
→ 이메일 인증
→ 로그인
→ JD·이력서 등록
→ 분석
→ 면접 1회 완료
→ 리포트 조회
→ 마이페이지 확인
```

기능별 완료 기준에는 다음 단계를 포함하겠습니다.

```text
코드 작성
→ 단위 테스트
→ API 검증
→ Frontend 연동
→ E2E
→ 실환경 검증
```

## Start–Stop–Continue

### Start — 새롭게 시작할 것

- 프로젝트 초기에 대표 E2E 시나리오부터 최소 기능으로 연결하겠습니다.
- API, DB, 상태값, 인증 방식의 변경 이유와 영향을 기록하겠습니다.
- 정상 흐름과 실패 시나리오를 기능 설계 단계에서 함께 정의하겠습니다.
- 기능별 Definition of Done에 통합·실환경 검증을 포함하겠습니다.

### Stop — 중단할 것

- 모든 문제를 직접 확인하고 해결하려는 방식을 줄이겠습니다.
- 문서와 코드의 차이를 프로젝트 후반에 한꺼번에 맞추지 않겠습니다.
- 재현 가능한 테스트 없이 한 번 성공한 것을 검증 완료로 판단하지 않겠습니다.
- 범위가 넓어졌을 때 우선순위 조정 없이 기능을 계속 추가하지 않겠습니다.

### Continue — 계속 유지할 것

- 전체 사용자 흐름을 기준으로 기능을 바라보는 태도
- 문제가 발생했을 때 원본과 현재 상태를 먼저 보존하는 방식
- 테스트 결과와 재현 가능한 근거로 상태를 판단하는 방식
- 복잡한 내용을 명세와 체크리스트 형태로 구조화해 공유하는 강점

## 프로젝트를 마치며

이번 프로젝트는 많은 기능을 구현하는 것보다 하나의 서비스를 끝까지 완성하기 위해 무엇이 필요한지를 경험한 프로젝트였습니다.

작은 API 불일치 하나도 전체 사용자 흐름을 중단시킬 수 있었고, 코드뿐 아니라 데이터 구조, 테스트, 예외 처리, 문서화, 배포, 팀원 간 합의가 함께 맞아야 실제 서비스가 완성된다는 점을 배웠습니다.

가장 크게 성장한 부분은 무엇을 구현했는가와 그것이 실제로 검증되었는가를 구분하게 된 것입니다. 앞으로는 프로젝트 초반부터 최소 E2E 흐름을 연결하고 명확한 완료 기준과 API 계약을 먼저 합의하되, 모든 문제를 직접 처리하기보다 팀원에게 책임과 검증 기준을 명확하게 전달하는 협업 방식을 발전시키고자 합니다.

> 좋은 기능을 많이 만드는 것보다, 사용자가 처음부터 끝까지 실제로 사용할 수 있는 하나의 흐름을 완성하는 것이 더 중요하다.
