## 김지윤

### 성과·기여·성장

이번 프로젝트에서 저는 질문 생성, 답변 충분성 판단, 약점 태깅, 꼬리질문 생성으로 이어지는 AI 면접 흐름을 담당했습니다. 사용자가 등록한 이력서, 자기소개서, JD, 프로젝트 정보를 바탕으로 질문을 생성하고, 답변이 질문 의도에 충분히 대응했는지 판단한 뒤 부족한 요소가 있을 때만 꼬리질문이 이어지도록 구현했습니다.

MVP 단계에서는 페르소나가 DB와 연결되기 전이었기 때문에 코치형, 실무형, 검증형 면접관이 각각 다른 방식으로 질문하고 답변을 검증하도록 코드와 프롬프트를 관리했습니다. 이후 관리자 담당자가 구현한 DB 기반 관리 구조와 실제 면접 흐름이 연결되는 과정도 함께 확인했습니다.

프로젝트 초반에는 제가 담당한 기능을 구현하는 데 대부분의 시간을 사용했습니다. MVP 범위가 많았기 때문에 중간 발표 전까지는 제 기능을 완성하고 관련 테스트를 통과시키는 것이 가장 중요하다고 생각했습니다. 개별 기능이 정상적으로 동작하고 테스트가 통과하면 제 역할은 어느 정도 끝났다고 생각했던 것 같습니다.

하지만 중간 발표 전 통합 테스트를 진행하면서 생각이 달라졌습니다. 개별 기능은 정상적으로 동작했지만, 다른 담당자의 기능과 연결되는 지점에서 여러 오류가 발생했습니다. 질문은 생성되지만 실제 면접 세션에서 정상적으로 사용되지 않거나, 답변 저장 이후 충분성 판단과 꼬리질문이 예상한 순서대로 이어지지 않는 문제가 있었습니다.

이때 기능 하나를 잘 만드는 것보다 그 기능이 앞뒤 단계와 자연스럽게 연결되도록 만드는 일이 더 중요하다는 것을 알게 되었습니다. 코드가 존재하고 단위 테스트가 통과하더라도 실제 사용자가 처음부터 끝까지 사용할 수 없다면 서비스 관점에서는 완성된 기능이라고 보기 어렵다는 점을 체감했습니다.

그래서 중간 발표 이후에는 새로운 고도화 기능을 계속 추가하기보다 기존 기능의 안정화와 연결 상태를 확인하는 데 더 집중했습니다. 충분성 판단이 불필요하게 반복 호출되지 않는지, 꼬리질문이 과도하게 생성되지 않는지, OpenAI 호출이 실패했을 때 Mock 결과로 조용히 대체되지 않는지, 예상 질문과 실제 면접 세션이 정상적으로 이어지는지 등을 점검했습니다.

처음에는 기능의 개수나 구현 속도가 프로젝트의 완성도를 결정한다고 생각했지만, 프로젝트 후반에는 사용자가 면접을 시작하고 답변한 뒤 리포트를 확인할 때까지 오류 없이 흐름을 완료할 수 있는지가 더 중요한 기준이 되었습니다.

또한 중간 발표와 최종 발표를 맡으면서 프로젝트 전반에 대한 이해도를 높이기 위해 노력했습니다. 초반에는 제 기능 구현에 집중한 나머지 다른 팀원들이 어떤 구조와 방식으로 개발했는지 충분히 알지 못했습니다. 발표 자료를 준비하려면 서비스 전체 흐름과 각 기능의 연결 관계를 설명할 수 있어야 했기 때문에, 중간 발표 이후에는 다른 팀원들의 코드와 구현 내용을 확인하는 시간을 가졌습니다.

코드 리뷰와 통합 흐름 점검을 통해 자료 분석 결과가 질문 생성에 어떻게 사용되는지, 답변 데이터가 평가와 리포트로 어떻게 전달되는지, 관리자 기능과 페르소나 설정이 면접 로직에 어떻게 반영되는지 등을 이해하게 되었습니다.

이 과정을 통해 빠르게 기능을 구현하는 것만큼, 어떤 방식으로 개발이 진행되었는지 이해하고 다른 팀원의 코드와 어떻게 연결되는지를 파악하는 것이 중요하다는 점을 배웠습니다. 특정 기능의 내부 로직만 알고 있는 것보다 전체 서비스에서 그 기능이 어떤 역할을 하는지 설명할 수 있어야 문제 발생 시 원인을 더 빠르게 찾을 수 있었습니다.

발표 경험도 개인적으로 의미가 있었습니다. 프로젝트의 배경과 기능, 기술 구조를 여러 사람 앞에서 정리해 말하는 연습을 할 수 있었고, 심사위원의 질문에 답하면서 제가 알고 있는 내용을 짧은 시간 안에 구조화해 전달하는 경험도 했습니다. 발표 당시에는 긴장도 많이 했지만, 결과적으로는 면접에서 프로젝트 경험을 설명하고 질문에 대응하는 연습을 간접적으로 해볼 수 있었습니다.

프로젝트 후반에는 여러 팀원이 비슷한 시기에 프로젝트에서 이탈하면서 부담을 많이 느꼈습니다. 기존 담당자가 작성한 코드의 흐름을 다시 확인해야 했고, 남은 일정 안에서 기능 안정화와 발표 준비를 함께 진행해야 했습니다. 모든 코드를 완벽하게 이해해야 한다는 부담도 있었지만, 반드시 확인해야 하는 핵심 흐름과 발표에 필요한 내용을 구분하며 마무리하려고 노력했습니다.

완성된 결과물이 처음 기획했던 모든 기능을 담고 있지는 않지만, 제한된 시간 안에서 사용자가 실제로 필요로 할 만한 기능이 무엇인지 팀원들과 고민하고, 최대한 하나의 서비스처럼 사용할 수 있는 결과물을 만들었다는 점에 만족합니다.

이번 프로젝트를 통해 AI 기능도 결국 하나의 서비스 기능이라는 점을 배웠습니다. 좋은 프롬프트만 작성한다고 기능이 완성되는 것이 아니라 입력 데이터, 세션 상태, 예외 처리, 저장 구조, 앞뒤 API가 모두 맞아야 실제 사용자에게 제공할 수 있습니다.

앞으로는 프로젝트 초반부터 제 담당 기능뿐 아니라 전체 사용자 흐름을 함께 확인하고 싶습니다. 새로운 기능을 빠르게 추가하는 것보다 최소한의 E2E 흐름을 먼저 연결하고, 기능의 완료 기준에 다른 파트와의 연동과 예외 상황 검증까지 포함하고자 합니다.

### 5 Whys — 통합 과정에서 오류가 늦게 발견된 이유

#### 문제

각 기능의 개별 테스트는 통과했지만, 중간 발표 전 통합 테스트에서 기능 간 연결 오류가 다수 발견되었습니다.

1. **왜 통합 단계에서 오류가 발생했는가?**  
   각 기능이 서로 기대하는 요청 형식, 상태값, 데이터 구조가 완전히 일치하지 않았기 때문입니다.

2. **왜 연결 구조를 더 일찍 확인하지 않았는가?**  
   MVP 구현량이 많아 각자 담당 기능을 완성하는 것을 먼저 목표로 삼았기 때문입니다.

3. **왜 개별 기능의 완성을 우선했는가?**  
   기능별 테스트가 통과하면 통합도 자연스럽게 될 것이라고 생각했기 때문입니다.

4. **왜 전체 흐름을 충분히 이해하지 못했는가?**  
   초반에는 역할이 명확하게 나뉘어 있었고, 다른 팀원의 구현 방식과 코드까지 확인할 필요성을 크게 느끼지 못했기 때문입니다.

5. **근본적으로 무엇이 부족했는가?**  
   프로젝트 초반부터 대표 사용자 흐름을 기준으로 기능 간 계약을 확인하고, 정기적으로 통합 테스트를 진행하는 과정이 부족했습니다.

#### 개선 방향

다음 프로젝트에서는 담당 기능을 구현하기 전에 해당 기능의 앞뒤 단계를 먼저 확인하고자 합니다.

```text
이전 단계의 출력 확인
→ 담당 기능 구현
→ 다음 단계의 입력 형식 확인
→ 개별 테스트
→ 통합 테스트
→ 예외 상황 검증
```

또한 기능 완료 기준에 다음 항목을 포함하고자 합니다.

- 실제 화면과 사용자 흐름에서 호출되는지
- 앞뒤 기능과 데이터 형식이 일치하는지
- 정상 입력뿐 아니라 누락 값과 실패 상황도 처리되는지
- 담당자가 아닌 팀원도 흐름을 이해할 수 있도록 문서화되어 있는지
- 기능 추가 이후 기존 흐름에 문제가 없는지 회귀 테스트했는지

### 프로젝트를 마치며

이번 프로젝트에서 가장 크게 배운 점은 기능을 많이 구현하는 것보다, 여러 기능을 하나의 흐름으로 연결하는 것이 더 어렵고 중요하다는 점입니다.

처음에는 제가 맡은 기능을 잘 완성하는 것이 가장 중요하다고 생각했습니다. 하지만 프로젝트를 진행하면서 제 기능이 앞 단계의 데이터를 제대로 전달받고, 그 결과가 평가와 리포트까지 이어져야 비로소 사용자에게 의미가 있다는 것을 알게 되었습니다.

앞으로는 구현 속도와 기능의 개수뿐 아니라 전체 사용자 흐름, 기능 간 연결, 안정성과 검증 상태를 함께 보며 개발하고자 합니다. 또한 발표와 코드 리뷰를 통해 얻은 경험을 바탕으로, 제가 만든 기능을 구현하는 데서 끝나지 않고 다른 사람에게 구조와 의도를 설명할 수 있는 개발자가 되고 싶습니다.

> 개별 기능의 완성보다, 사용자가 처음부터 끝까지 이용할 수 있는 하나의 흐름을 만드는 것이 더 중요하다.
