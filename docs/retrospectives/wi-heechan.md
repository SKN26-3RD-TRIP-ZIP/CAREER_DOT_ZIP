# 위희찬

## STAR-L — 기능 구현에서 운영 안정화까지

### Situation

CAREER.zip은 지원자의 이력서, 자기소개서, 채용공고와 프로젝트 정보를 분석해 개인화된 AI 모의면접과 평가 리포트를 제공하는 서비스입니다.

저는 프로젝트 초반에는 **STT·TTS 기반 음성 면접 기능**을 중심으로 개발했고, 후반에는 배포와 테스트, 인증·약관, JD 데이터 연동 등 운영 환경에서 발견된 문제를 해결했습니다.

```text
마이크 점검
→ 질문 TTS 재생
→ 답변 녹음
→ Whisper STT 변환
→ 답변과 발화 지표 저장
→ 충분성 판단 및 꼬리질문 생성
→ 평가·리포트 연결
→ 운영 환경 배포와 검증
```

기능을 운영 환경에 올린 뒤에는 HTTPS, 서버 메모리, OpenAI 요청 지연, OAuth 호환성, 약관 동의, JD 데이터 누락처럼 로컬 개발 단계에서는 잘 드러나지 않았던 문제도 마주했습니다. 이를 해결하면서 담당 기능만 구현하는 것에서 서비스 전체 흐름을 추적하고 안정화하는 방향으로 역할이 넓어졌습니다.

### Task

- 면접 시작 전 마이크 권한, 장치와 실제 입력 상태를 점검하는 과정 구현
- OpenAI TTS와 Whisper STT를 활용한 음성 면접 흐름 구성
- STT 결과와 발화 시간, 침묵 시간, 긴 침묵 횟수를 답변 데이터로 저장
- 음성 답변을 충분성 판단, 꼬리질문, 평가와 리포트에 연결
- 관리자 프롬프트 버전을 실제 면접 흐름에서 검증할 수 있는 테스트 기능 구현
- 외부 API 및 브라우저 기능 실패 시 재시도와 대체 경로 제공
- HTTPS, 메모리 부족과 외부 API 지연 등 운영 장애 원인 파악 및 안정화
- 인증·온보딩·약관과 JD 데이터 흐름에서 발생한 통합 오류 해결

### Action

#### 1. 음성 면접을 개별 기능이 아닌 하나의 파이프라인으로 구현했습니다

먼저 STT·TTS 모델을 비교하는 스크립트를 작성해 음성 면접에 사용할 방식을 검토했습니다. 이후 브라우저의 `MediaRecorder`로 녹음한 `webm` 파일을 Whisper에 전달하고, 변환된 텍스트를 기존 답변 저장 구조와 연결했습니다.

질문은 OpenAI TTS로 재생하되 중복 재생을 방지하고 항상 현재 질문만 들려주도록 제어했습니다. TTS 요청이 실패할 경우 브라우저 기본 음성 합성을 사용할 수 있도록 대체 경로도 마련했습니다.

음성 면접 시작 전에는 `getUserMedia`와 Web Audio API를 이용해 다음 항목을 확인했습니다.

- 브라우저의 마이크 기능 지원 여부
- 마이크 권한 허용 여부
- 사용 가능한 입력 장치
- 실제 음성 입력 레벨

단순히 권한만 허용된 상태가 아니라 일정 시간 정상적인 입력이 확인된 경우에만 음성 면접을 시작할 수 있도록 했습니다.

#### 2. STT 결과를 평가 가능한 데이터로 연결했습니다

Whisper의 word timestamp를 활용해 답변 텍스트뿐 아니라 다음과 같은 발화 지표도 계산했습니다.

- 실제 발화 시간
- 전체 침묵 시간
- 긴 침묵 횟수
- 처리 시간과 단어별 구간 정보

기존 평가와 리포트가 `answer_text`를 기준으로 동작하고 있었기 때문에 STT 결과를 `answer_text`와 `stt_text`에 함께 저장했습니다. 평가 단계에서는 음성 면접일 때 `stt_text`를 우선 사용하도록 연결해 기존 텍스트 면접 구조를 크게 변경하지 않으면서 음성 답변도 동일한 평가 흐름에 포함했습니다.

이후 답변 충분성 판단 결과에 따라 현재 질문 바로 뒤에 꼬리질문을 삽입하고, 마지막 질문까지 완료하면 세션 상태를 변경해 최종 리포트로 이어지도록 구성했습니다. URL의 세션 정보와 화면 상태가 어긋나는 문제도 발견해 세션 동기화 방식을 보완했습니다.

#### 3. 실패 단계에 맞는 재시도 방식을 적용했습니다

음성 면접은 마이크 권한, 녹음, 네트워크, STT, TTS와 답변 저장 등 실패할 수 있는 지점이 많았습니다. 모든 과정을 처음부터 반복하지 않도록 실패 위치에 따라 재시도 범위를 구분했습니다.

- STT 변환 전에 실패하면 원본 음성으로 STT부터 다시 요청
- STT 성공 후 저장에 실패하면 기존 변환 결과를 재사용
- TTS 재생에 실패하면 브라우저 기본 TTS 활용
- 답변이 너무 짧으면 저장을 막기보다 보완 안내 제공

이 과정을 통해 외부 API 연동은 정상 응답을 처리하는 것뿐 아니라 실패 사실을 사용자에게 어떻게 전달하고 어디부터 다시 시작하게 할지까지 포함해야 한다는 점을 배웠습니다.

#### 4. 관리자 프롬프트를 실제 면접으로 검증할 수 있게 했습니다

관리자가 프롬프트 문구와 버전을 확인하는 데서 끝나지 않고, 선택한 프롬프트 버전으로 실제 면접 흐름을 실행해볼 수 있는 테스트 기능을 Backend와 Frontend에 연결했습니다.

프롬프트는 화면에 저장되었다고 해서 품질이 보장되지 않습니다. 실제 지원자 데이터와 면접 질문 생성 흐름에서 결과를 확인해야 말투, 질문 깊이와 페르소나 반영 여부를 판단할 수 있었습니다. 이를 통해 관리 기능도 단순 CRUD가 아니라 운영자가 변경 결과를 안전하게 검증할 수 있는 과정까지 제공해야 한다는 점을 배웠습니다.

#### 5. 운영 장애를 서비스 전체의 문제로 보고 해결했습니다

배포 이후에는 로컬에서 확인하기 어려웠던 문제들이 발생했습니다.

- Frontend Nginx의 HTTPS 설정 복구
- OpenAI 리포트 평가 요청이 길어질 때 서버 자원을 계속 점유하는 문제 완화
- 작은 운영 서버에서 SBERT 모델 로딩으로 메모리가 부족해지는 문제 대응
- Git submodule의 수정 커밋을 메인 저장소에 반영하고 동일한 버전으로 배포
- 배포 후 health check를 통한 서비스 상태 확인

OpenAI 평가 요청에는 실행 범위를 제한하고, SBERT는 운영 환경의 자원 조건을 고려해 메모리 사용을 줄였습니다. 또한 배포 이미지를 커밋 SHA로 고정하고 헬스체크를 통과한 경우에만 성공으로 판단하도록 구성했습니다.

이를 통해 개발 환경에서 좋은 성능을 내는 기술이 운영 환경에서도 항상 적합한 것은 아니며, 서버 자원과 실패 비용까지 고려해 선택해야 한다는 점을 배웠습니다.

#### 6. 담당 영역 밖의 통합 오류도 사용자 흐름을 기준으로 해결했습니다

운영 환경을 점검하는 과정에서 음성 면접 외의 문제도 발견했습니다.

- 분석 페이지에 인증 보호 적용
- OAuth callback 실패 시 원인을 확인할 수 있도록 오류 처리 보강
- 소셜 로그인 사용자가 기존 사용자 모델과 호환되지 않던 문제 수정
- 기존 사용자의 온보딩 버전 누락 보완
- 일반 계정과 소셜 계정의 인증 완료 및 약관 동의 흐름 통합
- 약관 상세 내용과 재동의 안내 문구 개선
- JD 수정 시 구조화된 필드가 사라지는 문제 해결
- JD 목록 API와 관리 화면에서 구조화된 데이터를 다시 표시

이 문제들은 각기 다른 기능처럼 보였지만 모두 사용자가 로그인하고, 약관에 동의하고, 지원 자료를 입력한 뒤 면접에 진입하는 하나의 흐름을 막고 있었습니다. 담당 파일의 경계보다 사용자가 실제로 어디에서 중단되는지를 기준으로 문제를 추적했습니다.

### Result

- 질문 TTS, 마이크 점검, 답변 녹음, Whisper STT와 답변 저장을 연결한 음성 면접 흐름을 구현했습니다.
- 발화 시간과 침묵 지표를 저장해 음성 답변을 평가와 리포트에서 활용할 수 있는 기반을 마련했습니다.
- 충분성 판단과 꼬리질문, 세션 완료까지 이어지는 실제 사용자 흐름을 연결했습니다.
- 관리자 프롬프트 버전을 실제 면접 흐름에서 검증할 수 있는 테스트 기능을 추가했습니다.
- 단계별 재시도와 TTS 대체 경로를 마련해 외부 API 실패 상황의 사용자 경험을 보완했습니다.
- HTTPS, OpenAI 요청 지연과 SBERT 메모리 부족 문제를 해결하며 운영 환경의 안정성을 높였습니다.
- OAuth, 약관, 온보딩과 JD 데이터 오류를 수정해 면접 진입 전후의 사용자 흐름을 보완했습니다.

가장 큰 성과는 역할의 변화였습니다. 초반에는 STT·TTS 기능을 정상적으로 구현하는 데 집중했다면, 후반에는 인증부터 데이터 입력, 면접, 평가, 배포까지 전체 흐름을 추적하며 운영 문제를 해결할 수 있게 되었습니다.

### Learned

- **연동 기능의 완료 기준은 개별 API 성공이 아니라 전체 흐름의 완주입니다.** STT와 TTS가 동작해도 답변 저장과 평가, 리포트까지 이어지지 않으면 사용자에게는 완성된 기능이 아닙니다.
- **외부 API는 실패를 전제로 설계해야 합니다.** 재시도 범위, timeout, fallback, 중복 요청 방지와 오류 안내를 함께 정의해야 합니다.
- **운영 환경은 기술 선택의 조건입니다.** 모델의 성능뿐 아니라 서버 메모리, 응답 시간과 장애 시 영향까지 고려해야 합니다.
- **데이터 계약의 작은 불일치가 전체 흐름을 멈출 수 있습니다.** 인증 상태, 약관 버전과 JD 필드 하나가 누락되어도 사용자는 면접까지 도달하지 못할 수 있습니다.
- **배포와 QA는 마지막 부가 작업이 아닙니다.** 실제 환경에서 실행하고 관찰할 수 있어야 기능 완료 여부를 판단할 수 있습니다.

## 5 Whys — 로컬에서 동작한 서비스가 운영 환경에서 불안정했던 이유

### 문제

개발 환경에서는 각 기능이 정상적으로 동작했지만 운영 환경에서는 HTTPS, 메모리 부족, 외부 API 지연과 인증·데이터 연결 문제가 반복적으로 발생했습니다.

1. **왜 운영 환경에서 문제가 반복되었는가?**
   로컬 환경과 운영 서버의 HTTPS 구성, 메모리 용량, 환경변수와 외부 API 조건이 달랐기 때문입니다.

2. **왜 환경 차이가 서비스 장애로 이어졌는가?**
   SBERT 모델 로딩과 OpenAI 요청처럼 자원 사용량과 실행 시간이 큰 작업을 개발 환경의 정상 결과를 기준으로 연결했기 때문입니다.

3. **왜 운영 조건을 더 일찍 반영하지 못했는가?**
   기능 구현과 개별 API 연동을 우선했고, 실제 서버 자원과 HTTPS·OAuth를 포함한 통합 검증이 후반에 이루어졌기 때문입니다.

4. **왜 통합 검증이 후반에 이루어졌는가?**
   기능 완료 기준이 코드 작성과 로컬 동작에 집중되어 있었고, 운영 환경의 health check와 자원 사용량, 실패 복구가 명시적으로 포함되지 않았기 때문입니다.

5. **근본적으로 무엇이 부족했는가?**
   프로젝트 초기부터 최소 사용자 흐름을 운영과 유사한 환경에 배포하고 관찰하는 기준이 부족했습니다.

### 개선 방향

다음 프로젝트에서는 핵심 기능을 모두 만든 뒤 배포하지 않고, 최소 흐름을 먼저 운영과 유사한 환경에 올려 확인하겠습니다.

```text
로그인·약관 동의
→ JD·지원 자료 입력
→ 면접 세션 생성
→ 질문 1개 재생
→ 답변 녹음 및 STT
→ 평가 결과 확인
→ health check와 자원 사용량 확인
```

기능별 완료 기준에도 다음 항목을 포함하겠습니다.

```text
로컬 동작
→ 단위·통합 테스트
→ 외부 API 실패 테스트
→ 운영 유사 환경 배포
→ 로그·메모리·응답 시간 확인
→ 복구와 롤백 검증
```

## Start–Stop–Continue

### Start — 새롭게 시작할 것

- 프로젝트 초기에 최소 E2E 흐름을 운영과 유사한 환경에 배포하겠습니다.
- 외부 API 연동 전에 성공, 실패, timeout과 재시도 시나리오를 정의하겠습니다.
- 모델을 선택할 때 정확도뿐 아니라 메모리와 응답 시간도 함께 측정하겠습니다.
- 브라우저별 마이크 권한과 녹음 형식 차이를 초기부터 테스트하겠습니다.
- 로그와 상태값으로 사용자가 어느 단계에서 실패했는지 추적할 수 있게 하겠습니다.

### Stop — 중단할 것

- 정상 입력에서 한 번 동작한 결과만으로 기능이 완료되었다고 판단하지 않겠습니다.
- Mock 성공과 실제 외부 API 및 운영 환경의 성공을 같은 상태로 취급하지 않겠습니다.
- 개발 장비에서 동작한다는 이유로 운영 서버의 자원 조건을 뒤로 미루지 않겠습니다.
- API 필드와 인증 상태의 불일치를 화면에서 임시로 우회하지 않겠습니다.

### Continue — 계속 유지할 것

- 복잡한 기능을 녹음, 변환, 저장, 평가처럼 작은 단계로 나누어 해결하는 방식
- 기존 데이터 구조와의 호환성을 고려해 새로운 기능을 연결하는 태도
- 실패 시 전체를 반복하기보다 필요한 단계부터 복구하도록 설계하는 방식
- 담당 파일에 한정하지 않고 사용자의 전체 흐름을 따라 문제를 추적하는 태도
- 운영 문제를 로그와 재현 가능한 근거로 좁혀가는 방식
- 구현 후 코드 리뷰와 QA를 통해 개선점을 다시 찾는 습관

## 프로젝트를 마치며

이번 프로젝트를 통해 음성 면접은 단순한 STT·TTS 기능이 아니라 브라우저 장치, 외부 AI API, 데이터 저장, 평가 로직과 사용자 경험이 함께 맞물리는 통합 기능이라는 점을 배웠습니다.

또한 로컬에서 잘 동작하는 코드를 만드는 것과 실제 사용자가 안정적으로 사용할 수 있는 서비스를 만드는 것은 다른 과제였습니다. HTTPS, OAuth, 서버 메모리와 API 지연 문제를 해결하면서 기능 개발뿐 아니라 운영 환경의 제약과 전체 사용자 흐름까지 함께 고려하게 되었습니다.

가장 크게 성장한 부분은 문제가 발생했을 때 담당 기능 안에서만 원인을 찾지 않고, 인증부터 데이터 입력, 면접, 평가와 배포까지 흐름을 따라가며 원인을 좁힐 수 있게 된 것입니다. 앞으로는 전체 흐름을 단계별로 나누는 강점을 유지하면서 초기 설계부터 실패 시나리오, 데이터 정합성, 운영 자원과 검증 기준까지 함께 정의하는 개발자가 되고자 합니다.

> 기능을 만드는 것에서 끝나지 않고, 실제 환경에서 실패해도 원인을 찾고 다시 이어갈 수 있는 서비스를 만들자.
